# 디펜스 발표 대본

각 페이지당 약 40초 내외

## 1페이지 - 제목

안녕하세요. 저는 "Ghidra 기반 큐브샛 펌웨어 에뮬레이션 및 정적 분석 보강 프레임워크"라는 주제로 발표를 진행하겠습니다.

본 연구의 핵심 문제의식은 큐브샛 펌웨어를 분석할 때, 바이너리 정적 분석만으로는 실제 실행 중 형성되는 제어 흐름을 모두 파악하기 어렵다는 점입니다.

특히 큐브샛의 온보드 컴퓨터 펌웨어는 타이머, 통신 장치, 전력 장치 같은 주변장치와 계속 값을 주고받으면서 동작합니다. 따라서 본 연구에서는 Ghidra 안에서 펌웨어를 에뮬레이션하고, 실행 중 관찰한 정보를 다시 정적 분석 결과에 반영하는 방법을 제안합니다.

## 2페이지 - 큐브샛이란?

먼저 큐브샛이 무엇인지 간단히 설명드리겠습니다.

큐브샛은 10cm 정육면체를 기본 단위로 하는 표준화된 소형 위성 규격입니다. 이 기본 단위를 1U라고 부르고, 실제 큐브샛은 임무에 따라 1U, 2U, 3U 또는 그 이상의 크기로 구성될 수 있습니다.

큐브샛은 크기가 작고 개발 비용이 비교적 낮기 때문에 교육, 연구, 기술 실증, 상업 임무 등에 널리 사용됩니다. 하지만 본 연구에서 중요한 점은, 큐브샛도 결국 하나의 임베디드 시스템이라는 것입니다. 즉 제한된 하드웨어 위에서 펌웨어가 실제 장치들과 상호작용하며 위성 전체의 동작을 제어합니다.

## 3페이지 - 큐브샛과 다른 임베디드 시스템의 차이

그렇다면 큐브샛은 일반적인 임베디드 시스템과 무엇이 다를까요?

가장 큰 차이는 발사 이후에는 물리적으로 접근해서 수리하거나 디버깅하기가 거의 불가능하다는 점입니다. 이 때문에 큐브샛에서는 오류가 발생했을 때 전체 시스템이 한 번에 멈추지 않도록 소프트웨어적인 fault tolerance가 매우 중요합니다.

그래서 하나의 MCU가 모든 기능을 처리하기보다는, OBC, 전력계, 통신계, 자세 제어계처럼 여러 서브시스템으로 나누어 구성하는 경우가 많습니다. 또한 발사 전에 가능한 한 버그를 많이 찾아야 하고, 이때 바이너리 기반 정적 분석은 펌웨어 구조와 호출 관계를 이해하는 중요한 방법이 됩니다.

## 4페이지 - 큐브샛의 일반적인 구조

이 그림은 큐브샛의 일반적인 구조를 OBC 중심으로 단순화해서 나타낸 것입니다.

가운데 OBC에는 CPU가 있고, 타이머, 인터럽트 컨트롤러, DMA 컨트롤러, I2C 컨트롤러 같은 내부 장치들이 함께 존재합니다. 그리고 바깥쪽에는 전력계, 배터리, 자세 제어계, 통신계처럼 기능별 서브시스템이 연결됩니다.

본 발표에서는 OBC 펌웨어가 직접 접근하거나 통신을 통해 값을 주고받는 하드웨어들을 주변장치라고 부르겠습니다. 실제 보드마다 장치 구성과 이름은 달라질 수 있지만, OBC 펌웨어가 여러 주변장치 상태에 의존해서 실행된다는 구조는 공통적으로 나타납니다.

## 5페이지 - 큐브샛 OBC 펌웨어 특징

이제 OBC 펌웨어의 특징을 조금 더 보겠습니다.

OBC 펌웨어는 단순히 CPU 명령어만 순서대로 실행하는 프로그램이 아니라, 실행 중에 계속 주변장치와 상호작용합니다. 예를 들어 온도 센서에서 특정 부분의 온도가 너무 낮다는 값이 들어오면, 펌웨어는 그 값을 읽고 히터를 켜는 명령을 내릴 수 있습니다.

또 통신 장치에서 지상국 명령이 수신되었는지 확인하고, 전력계나 자세 제어계의 상태를 읽어서 다음 동작을 결정하기도 합니다. 즉 OBC 펌웨어의 실제 제어 흐름은 CPU 내부 상태뿐 아니라, 주변장치가 어떤 값을 반환하고 어떤 인터럽트를 발생시키는지에 따라 달라집니다.

## 6페이지 - 큐브샛 OBC 펌웨어 정적 분석의 한계

이런 주변장치 의존성 때문에 OBC 펌웨어를 정적 분석만으로 분석할 때 한계가 생깁니다.

정적 분석은 프로그램을 실행하지 않고 코드 구조를 분석합니다. 그래서 직접 호출이나 명시적인 분기처럼 코드 안에 대상이 드러나는 흐름은 비교적 잘 복원할 수 있습니다. 하지만 주변장치의 인터럽트가 발생해야 호출되는 함수나, 주변장치 레지스터 값이 특정 조건을 만족해야 실행되는 경로는 바이너리만 보고는 정확히 파악하기 어렵습니다.

소스 코드가 있으면 설계 의도나 타입 정보를 더 활용할 수 있지만, 실제로 위성에서 실행되는 것은 최종 바이너리입니다. 따라서 OBC 펌웨어 바이너리만으로는 주변장치에 의해 결정되는 제어 흐름을 완전히 복원하기 어렵다는 문제가 있습니다.

## 7페이지 - Ghidra 기반 정적 분석의 한계

이때 사용할 수 있는 대표적인 도구가 Ghidra입니다.

Ghidra는 오픈소스이고 무료로 공개된 바이너리 분석 도구이며, 역어셈블, 역컴파일, 심볼 분석, 참조 분석, 호출 그래프 생성 같은 기능을 제공합니다. 또 Ghidra는 각 아키텍처의 명령어 의미를 SLEIGH 명세로 정의하고, 이를 P-code라는 중간 표현으로 변환합니다.

이 점은 다양한 아키텍처의 펌웨어를 공통된 방식으로 분석할 수 있게 해주는 장점입니다. 하지만 Ghidra의 기본 정적 분석 역시 실제 주변장치의 상태나 인터럽트, I2C 응답을 자동으로 알지는 못합니다. 따라서 Ghidra를 사용하더라도 주변장치와의 상호작용에서 생기는 실제 실행 흐름은 분석 결과에 빠질 수 있습니다.

## 8페이지 - 아이디어: 에뮬레이션을 통한 정적 분석 결과 보강

그래서 본 연구의 핵심 아이디어는 에뮬레이션을 통해 정적 분석 결과를 보강하는 것입니다.

먼저 펌웨어를 실제 하드웨어 없이 실행해 보고, 실행 중에 관찰되는 동적 제어 흐름을 기록합니다. 그리고 기존 정적 분석 결과에 빠져 있던 간접 호출이나 간접 분기 정보를 이 실행 결과로 보완합니다.

Ghidra의 경우, 프로그램을 import하고 정적 분석을 수행하면 함수, 데이터 타입, 심볼, cross-reference 같은 정보가 프로그램 데이터베이스에 저장됩니다. 본 발표에서는 이런 분석 결과 저장소를 일반적으로 펌웨어 데이터베이스라고 부르겠습니다. 이 데이터베이스에 동적으로 관찰한 제어 흐름을 추가하면, 이후 정적 분석이 더 많은 호출 관계를 사용할 수 있게 됩니다.

## 9페이지 - 핵심 주장

이 슬라이드는 방금 설명한 아이디어를 한 문장으로 정리한 것입니다.

본 연구에서는 큐브샛 펌웨어를 실제 하드웨어 없이 에뮬레이션해서 실행 중 instruction trace를 얻습니다. 여기서 중요한 것은 단순히 실행했다는 사실이 아니라, 실행 중에 관찰되는 제어 흐름 정보를 기존 펌웨어 데이터베이스에 다시 반영한다는 점입니다.

즉 에뮬레이션 결과를 별도의 로그로만 남기는 것이 아니라, Ghidra가 이미 가지고 있는 함수와 참조 정보에 연결합니다. 이렇게 하면 정적 분석만으로는 도달하지 못했던 함수나 간접 호출 대상이 이후 분석 범위에 포함될 수 있습니다.

## 10페이지 - 프레임워크 개요

전체 프레임워크는 이 그림과 같이 구성됩니다.

먼저 펌웨어 바이너리는 정적 분석기를 통해 분석되고, 그 결과는 펌웨어 데이터베이스에 저장됩니다. 동시에 같은 펌웨어를 큐브샛 에뮬레이터 위에서 실행합니다. 이 에뮬레이터는 실행 중 간접 호출이나 간접 분기 같은 동적 제어 흐름을 로그로 남깁니다.

그 다음 데이터베이스 보강 모듈이 기존 펌웨어 데이터베이스와 동적 제어 흐름 로그를 결합합니다. 최종적으로는 보강된 펌웨어 데이터베이스가 만들어지고, 이후 호출 그래프 분석이나 도달 가능성 분석에서 더 넓은 범위를 확인할 수 있게 됩니다.

## 11페이지 - 큐브샛 에뮬레이터 구현 개요

이제 에뮬레이터 구현을 설명하겠습니다.

에뮬레이터 구현의 목표는 앞에서 본 큐브샛 OBC 중심 구조를 소프트웨어적으로 모방하는 것입니다. 즉 CPU는 명령어를 실행하는 에뮬레이션 엔진으로 대체하고, 타이머, 인터럽트 컨트롤러, DMA 컨트롤러, I2C 컨트롤러 같은 내부 주변장치는 소프트웨어 장치 모델로 구현합니다.

또 EPS, 배터리, ADCS, RF 송수신 장치 같은 외부 서브시스템은 통신 인터페이스를 통해 명령과 응답을 주고받는 장치로 모델링합니다. 여기서 목표는 실제 하드웨어를 주기 정확하게 복제하는 것이 아니라, 펌웨어가 실행 중 관찰하는 레지스터 값, 인터럽트, 통신 응답 같은 동작을 제공하는 것입니다.

## 12페이지 - 에뮬레이터를 통해 CPU 대체

먼저 CPU를 어떻게 대체할지 결정해야 했습니다.

임베디드 펌웨어 에뮬레이션에서는 QEMU가 널리 사용됩니다. QEMU는 빠르고 성숙한 에뮬레이터이지만, 대상 아키텍처와 보드를 지원해야 하고, 분석 결과를 Ghidra 데이터베이스와 연결하려면 별도의 연동 과정이 필요합니다.

반면 본 연구에서는 Ghidra script 기반의 에뮬레이터를 사용했습니다. Ghidra는 SLEIGH와 P-code를 통해 여러 아키텍처의 명령어 의미를 공통된 중간 표현으로 다룹니다. 따라서 특정 CPU 에뮬레이터를 새로 만드는 대신, Ghidra의 P-code 실행 체계를 활용할 수 있습니다. 또한 실행 결과를 같은 Ghidra 프로그램 데이터베이스에 바로 반영할 수 있다는 점이 본 연구의 목적과 잘 맞습니다.

## 13페이지 - MMIO 장치 구현

다음은 MMIO 장치 구현입니다.

OBC 내부 주변장치는 보통 메모리 주소 공간에 매핑된 레지스터를 통해 제어됩니다. 펌웨어가 특정 주소를 읽거나 쓰면, 단순한 메모리 접근이 아니라 장치 상태 변화나 인터럽트 발생 같은 부수 효과가 일어날 수 있습니다.

본 구현에서는 이러한 장치들을 MMIODevice와 MMIORegion 형태로 모델링했습니다. 예를 들어 PDCA의 특정 channel에서 interrupt enable register의 특정 bit를 1로 설정하면, 해당 channel의 interrupt mask register에도 그 상태가 반영됩니다. 또 control register의 transfer enable bit가 설정되면 실제 데이터 전송이 시작된 것으로 처리합니다. 이런 식으로 펌웨어가 기대하는 관찰 가능한 장치 동작을 제공합니다.

## 14페이지 - P-code 단위 메모리 접근 후킹

그렇다면 펌웨어가 MMIO 주소에 접근하는 순간을 어떻게 알 수 있을까요?

본 연구에서는 Ghidra의 P-code 실행 과정에서 메모리 접근을 후킹합니다. 명령어가 PcodeOp으로 변환되면, LOAD나 STORE 같은 연산을 관찰할 수 있습니다. 이때 접근 주소가 일반 메모리가 아니라 MMIO 장치 영역에 해당하는지 확인하고, 해당 장치 모델의 read 또는 write 로직을 호출합니다.

예를 들어 펌웨어가 PDCA control register에 값을 쓰면, 에뮬레이터는 그 주소가 PDCA 장치에 속한다는 것을 확인하고 pdca_enable과 같은 내부 동작을 수행합니다. 이 방식으로 단순 명령어 실행과 주변장치의 side effect를 연결할 수 있습니다.

## 15페이지 - 인터럽트 반영

다음은 인터럽트 반영입니다.

큐브샛 OBC 펌웨어에서는 타이머 인터럽트가 주기적인 작업 실행이나 컨텍스트 스위칭에 중요한 역할을 합니다. 따라서 CPU 명령어만 실행해서는 실제 펌웨어의 작업 흐름을 재현하기 어렵습니다.

본 구현에서는 OBC와 인터럽트 컨트롤러인 INTC를 연결하고, 각 명령어 실행 후 현재 처리 가능한 인터럽트가 있는지 확인합니다. 또한 타이머 카운터에 해당하는 TC를 별도 스레드로 구현하여 주기적으로 인터럽트가 발생하도록 했습니다. 이를 통해 펌웨어가 인터럽트 핸들러에 진입하고, 스케줄러에 의해 작업 문맥이 전환되는 흐름을 재현할 수 있습니다.

## 16페이지 - I2C 장치 모델링

다음은 I2C 장치 모델링입니다.

OBC는 전력계, 자세 제어계, RF 송수신 장치 같은 외부 서브시스템과 I2C 같은 통신 인터페이스를 통해 명령과 응답을 주고받습니다. 따라서 에뮬레이터에서도 각 I2C 장치가 펌웨어의 요청에 대해 적절한 응답을 반환할 수 있어야 합니다.

본 구현에서는 EPS, ADCS, VRX, UTX 같은 장치들을 I2C 장치 모델로 구현했습니다. 특히 VRX와 UTX는 RF module simulator와 연결하여 지상국과의 통신을 모델링했습니다. 외부 TCP 클라이언트가 명령 패킷을 보내면 VRX를 통해 OBC로 전달되고, 펌웨어가 생성한 응답은 UTX를 통해 다시 클라이언트로 전송됩니다.

## 17페이지 - I2C 통신 모델링

이 슬라이드는 I2C 통신이 에뮬레이터 안에서 어떻게 연결되는지를 보여줍니다.

펌웨어는 직접 I2C 장치 객체를 호출하는 것이 아니라, TWIM과 PDCA 같은 컨트롤러를 통해 데이터를 주고받습니다. 그래서 본 구현에서는 TWIM 및 PDCA를 통한 데이터 전송 흐름까지 모델링했습니다.

예를 들어 지상국에서 uplink 명령이 들어오면, 펌웨어의 uplink manager는 vrx_get_frames 명령을 통해 현재 수신된 프레임 개수를 확인할 수 있습니다. 이때 VRX 장치 모델은 내부 큐에 저장된 패킷 상태를 기반으로 응답을 반환합니다. 이후 펌웨어는 프레임을 읽고 명령을 해석한 뒤, 해당 핸들러를 호출하게 됩니다.

## 18페이지 - 동적 제어 흐름 로그 구현

이제 에뮬레이션 결과를 어떻게 정적 분석 보강에 사용할지 설명하겠습니다.

에뮬레이터는 각 명령어를 실행한 뒤, 해당 명령어가 indirect control flow를 만드는 명령어인지 확인합니다. Ghidra 내부 용어로는 FlowType이 computed call 또는 computed jump에 해당하는 경우입니다.

이런 명령어는 정적 분석만으로 대상 주소를 알기 어려운 경우가 많습니다. 하지만 에뮬레이션 중에는 실제 레지스터 값과 메모리 값이 존재하므로, 그 순간 어느 주소로 이동했는지 관찰할 수 있습니다. 본 구현에서는 이때 명령어 주소와 실제 목적지 주소를 동적 제어 흐름 로그로 기록합니다.

## 19페이지 - 동적 제어 흐름 로그와 펌웨어 데이터베이스

이 슬라이드는 동적 제어 흐름 로그가 전체 프레임워크 안에서 어디에 위치하는지를 보여줍니다.

기존 정적 분석기는 펌웨어를 분석해서 펌웨어 데이터베이스를 만듭니다. 하지만 정적 분석만으로는 간접 호출이나 주변장치 의존적인 흐름이 빠질 수 있습니다. 큐브샛 에뮬레이터는 같은 펌웨어를 실행하면서 실제로 관찰된 간접 제어 흐름을 별도 로그로 남깁니다.

즉 여기까지의 결과는 두 가지입니다. 하나는 Ghidra가 만든 기존 정적 분석 데이터베이스이고, 다른 하나는 에뮬레이션이 만든 동적 제어 흐름 로그입니다. 다음 단계에서는 이 둘을 연결해서 기존 데이터베이스를 보강합니다.

## 20페이지 - 동적 제어 흐름을 데이터베이스에 적용

동적 제어 흐름 로그는 외부 파일로만 남겨두면 이후 정적 분석에서 자동으로 사용되기 어렵습니다.

그래서 본 구현에서는 로그에 기록된 흐름을 Ghidra 프로그램 데이터베이스에 직접 반영했습니다. 구체적으로, 실행 중 관찰한 source 주소와 destination 주소가 서로 다른 함수 사이의 흐름이라면 Ghidra reference manager를 통해 참조를 추가합니다.

이렇게 추가된 참조는 이후 호출 그래프 생성이나 도달 가능성 분석에서 기존 정적 분석 결과와 함께 사용될 수 있습니다. 또한 필요하면 북마크 같은 메타데이터로 표시하여, 어떤 흐름이 에뮬레이션을 통해 관찰된 것인지 분석자가 확인할 수 있게 했습니다.

## 21페이지 - 케이스 스터디: RANDEV

이제 구현과 평가에 사용한 대상 펌웨어를 설명하겠습니다.

본 연구에서는 큐브샛 OBC 펌웨어 대상으로 RANDEV를 선택했습니다. RANDEV는 AVR32 기반 OBC 환경이며, 타이머, 인터럽트, PDCA/TWIM 기반 I2C 통신, RF 송수신 장치와의 명령-응답 흐름을 함께 사용합니다.

실제 구현에서는 RANDEV에 특화된 정보도 필요했습니다. 예를 들어 장치별 MMIO 주소 범위, PDCA와 TWIM 채널 매핑, I2C slave 주소, 각 장치의 명령-응답 형식 등을 설정해야 했습니다. 따라서 본 연구의 일반성은 모든 펌웨어를 수정 없이 바로 실행한다는 의미가 아니라, Ghidra P-code 실행 계층과 장치 모델 연결 방식, 데이터베이스 보강 구조를 다른 대상에도 재사용할 수 있다는 의미입니다.

## 22페이지 - 평가 질문

평가는 세 가지 질문을 중심으로 진행했습니다.

첫째, 구현한 에뮬레이터가 telemetry 명령에 대해 올바른 응답을 전송할 수 있는지 확인했습니다. 이는 실제 하드웨어 없이도 펌웨어의 명령-응답 흐름을 재현할 수 있는지를 보기 위한 질문입니다.

둘째, 에뮬레이션을 통해 기존 정적 분석 결과를 실제로 보강할 수 있는지 확인했습니다. 즉 기존 Ghidra 분석에서 대상이 불명확했던 간접 호출 중 얼마나 많은 대상이 실행 중 확인되었는지를 봤습니다.

셋째, 이렇게 보강된 결과를 사용했을 때 정적 분석 가능 범위가 기존보다 넓어지는지 확인했습니다. 이를 위해 보강 전후 호출 그래프에서 도달 가능한 함수 수를 비교했습니다.

## 23페이지 - 평가 1: telemetry 응답 검증

첫 번째 평가는 telemetry 명령에 대한 응답 검증입니다.

에뮬레이터가 실제로 의미 있는 실행 환경을 제공한다면, 펌웨어가 지상국 명령을 수신하고, 명령 핸들러를 실행한 뒤, 응답 패킷을 생성해서 송신할 수 있어야 합니다. 이를 확인하기 위해 펌웨어가 지원하는 각 I2C 장치에서 수행할 수 있는 명령을 보내고 응답을 비교했습니다.

비교 대상 명령은 총 43개였고, 이 중 42개에 대해서 기대한 응답과 일치했습니다. 불일치한 사례는 EPS의 eps_get_status 명령이었고, 이 경우 timeout이 발생했습니다. 즉 완전한 장치 재현은 아니지만, 대부분의 telemetry 명령에 대해 실제 명령-응답 흐름을 재현할 수 있음을 확인했습니다.

## 24페이지 - 평가 2: 정적 분석 결과 보강 여부

두 번째 평가는 에뮬레이션을 통해 기존 정적 분석 결과가 실제로 보강되었는지 확인하는 것입니다.

Ghidra 디컴파일 결과에서 확인된 ICALL은 총 66개였습니다. 이 중 콘솔 명령 계열 함수 8개는 본 구현에서 콘솔 부분을 모델링하지 않았기 때문에 평가 대상에서 제외했습니다.

남은 58개의 간접 호출 위치 중 16개에 대해서 하나 이상의 호출 대상이 실행 중 확인되었습니다. 그리고 이 정보는 단순 로그가 아니라 Ghidra 데이터베이스의 참조와 메타데이터로 반영되었습니다. 이 결과는 주변장치와 외부 입력을 모델링한 에뮬레이션이 정적 분석만으로는 불명확했던 간접 제어 흐름 일부를 실제 대상으로 연결할 수 있음을 보여줍니다.

## 25페이지 - 평가 3: 정적 분석 가능 범위 확장

세 번째 평가는 보강된 데이터베이스로 정적 분석을 수행했을 때 분석 가능 범위가 늘어나는지 확인하는 것입니다.

이를 위해 에뮬레이션 전후의 펌웨어 데이터베이스에 대해 동일한 로직으로 call graph를 생성했습니다. Ghidra에서 디컴파일된 함수 각각을 node로 보고, main entrypoint와 task별 entrypoint에서부터 정적으로 도달 가능한 node 수를 계산했습니다.

콘솔 관련 함수는 앞선 평가와 동일하게 제외했습니다. 그 결과 전체 함수 953개 중 에뮬레이션 이전에는 744개가 도달 가능했고, 데이터베이스 보강 이후에는 792개가 도달 가능했습니다. 즉 48개의 함수가 새롭게 호출 그래프 분석 범위에 들어왔고, 이는 이후 정적 분석에 반영되는 함수의 수가 늘어났음을 보여줍니다.

## 26페이지 - 논의

이 결과를 해석할 때 논의해야 할 한계도 있습니다.

첫 번째는 다른 펌웨어에도 적용 가능한가 하는 점입니다. 현재 구현에는 RANDEV에 특화된 정보가 포함되어 있습니다. 예를 들어 MMIO 주소 범위, 인터럽트 맵, I2C slave 주소, 장치별 명령 형식은 대상 보드마다 달라집니다. 따라서 다른 펌웨어에 적용하려면 board description이나 장치 모델을 새로 제공해야 합니다.

두 번째는 속도 문제입니다. Ghidra 기반 에뮬레이터는 QEMU보다 느립니다. 하지만 본 연구의 목적은 고속 실행이 아니라, PcodeOp 단위로 MMIO 접근과 간접 제어 흐름을 관찰하고 이를 Ghidra 데이터베이스에 반영하는 것입니다. 따라서 분석용 에뮬레이터로서는 감수 가능한 절충점으로 볼 수 있습니다.

## 27페이지 - 결론

마지막으로 결론입니다.

본 연구는 큐브샛 펌웨어를 실제 하드웨어 없이 에뮬레이션하고, 실행 중 얻은 instruction trace와 동적 제어 흐름 정보를 기존 펌웨어 데이터베이스에 반영하는 프레임워크를 제안했습니다.

구현 측면에서는 Ghidra P-code 실행 엔진을 기반으로 CPU 실행을 대체하고, MMIO 장치, 인터럽트, I2C 통신, RF 명령-응답 흐름을 모델링했습니다. 평가에서는 telemetry 명령 43개 중 42개에 대해 응답을 재현했고, 간접 호출 58개 중 16개에서 대상을 확인했으며, 호출 그래프에서 도달 가능 함수 수가 744개에서 792개로 증가했습니다.

결론적으로 본 연구는 동적 실행 정보를 Ghidra 정적 분석 데이터베이스와 결합함으로써, 주변장치 의존적인 큐브샛 펌웨어의 분석 범위를 확장할 수 있음을 보였습니다.

## 28페이지 - Backup Slides

여기서부터는 백업 슬라이드입니다.

본 발표의 본문에서는 문제 배경, 프레임워크 설계, 구현, 평가 결과를 중심으로 설명했습니다. 이후 슬라이드들은 주로 질문이 나올 수 있는 지점에 대한 보충 설명입니다.

예를 들어 문제 정의를 기존 QEMU나 Ghidra와 비교해서 다시 설명하거나, 구현 요소를 더 세분화해서 보여주거나, 전체 프레임워크를 다른 관점의 그림으로 다시 정리하는 내용입니다. 질의응답에서 특정 부분을 더 자세히 설명할 필요가 있을 때 이 슬라이드들을 사용하겠습니다.

## 29페이지 - Problem Statement 관련

이 슬라이드는 본 연구의 문제 정의를 기존 도구와 비교해서 다시 정리한 것입니다.

먼저 에뮬레이터를 구현해서 테스팅이나 펌웨어 실행이 쉬워졌느냐는 질문이 있을 수 있습니다. QEMU와 비교하면, 주변장치 모델링은 어차피 필요합니다. 다만 Ghidra 기반 접근에서는 펌웨어가 이미 Ghidra의 P-code 표현으로 올라오기 때문에, 실행 결과를 Ghidra 데이터베이스와 연결하는 단계가 자연스럽습니다.

더 중요한 질문은 분석이 쉬워졌느냐입니다. 기존 Ghidra는 정적 분석에는 강하지만 주변장치와의 상호작용을 분석 결과에 직접 반영하지는 않습니다. 본 연구의 기여는 이 부분입니다. 주변장치 모델을 통해 실행 흐름을 일부 재현하고, 그 결과를 정적 분석 데이터베이스에 반영할 수 있게 했습니다.

## 30페이지 - 분석 및 테스팅이 어려운 이유

이 슬라이드는 큐브샛 OBC 펌웨어의 분석과 테스팅이 왜 어려운지 기존 도구 관점에서 정리한 것입니다.

테스팅 관점에서는 QEMU 같은 도구를 사용할 수 있지만, 대상 아키텍처나 보드가 지원되지 않으면 많은 부분을 직접 구현해야 합니다. 특히 큐브샛 펌웨어는 MMIO, 인터럽트, DMA, I2C 장치처럼 주변장치 의존성이 크기 때문에 장치 모델링이 필수입니다.

분석 관점에서는 Ghidra가 강력한 정적 분석 기능을 제공하지만, 실행 환경 자체를 자동으로 만들어 주지는 않습니다. 즉 바이너리 구조는 볼 수 있지만, 주변장치 응답이나 인터럽트에 의해 실제로 형성되는 제어 흐름은 빠질 수 있습니다. 본 연구는 이 두 문제 사이에서, Ghidra 안의 실행과 정적 분석 보강을 연결하는 접근입니다.

## 31페이지 - OBC 펌웨어 분석이 어려운 이유

이 슬라이드는 문제의 핵심을 더 짧게 다시 말한 것입니다.

OBC 펌웨어는 주변장치와 계속 상호작용하고, 그 주변장치의 behavior에 dependent한 control flow를 가집니다. 예를 들어 타이머나 통신 장치의 인터럽트가 발생해야만 호출되는 함수가 있을 수 있고, 특정 레지스터 값이 설정되어야만 루프를 빠져나가거나 다음 단계로 진행하는 코드가 있을 수 있습니다.

바이너리만 놓고 보면 이런 값들이 실제 실행 중 어떤 상태가 되는지 알기 어렵습니다. 특히 소스 코드가 아니라 최종 바이너리를 대상으로 분석할 때는 함수 포인터, 간접 호출, 명령 디스패처 같은 구조가 더 불명확해질 수 있습니다. 그래서 본 연구에서는 주변장치 상태를 에뮬레이션으로 제공하고, 그 결과를 정적 분석에 다시 반영합니다.

## 32페이지 - 큐브샛 에뮬레이터 구현 세부 항목

이 슬라이드는 에뮬레이터 구현을 다섯 가지 항목으로 나누어 정리한 것입니다.

첫째, MMIO 장치 구현입니다. 펌웨어가 특정 주소를 읽고 쓸 때 장치 상태가 바뀌도록 모델링했습니다. 둘째, P-code 단위 메모리 접근 후킹입니다. Ghidra 실행 중 LOAD와 STORE를 관찰해서 MMIO 접근을 장치 모델로 전달합니다.

셋째, 인터럽트 반영입니다. TC와 INTC를 모델링해서 타이머 인터럽트와 컨텍스트 스위칭 흐름을 재현했습니다. 넷째, I2C 장치 모델링 및 연결입니다. EPS, ADCS, VRX, UTX 같은 외부 장치가 명령에 응답하도록 했습니다. 마지막으로 RF module simulator를 구현해서 TCP 클라이언트를 통해 지상국 명령과 원격 측정 응답 흐름을 테스트할 수 있게 했습니다.

## 33페이지 - 큐브샛 에뮬레이션을 통한 메타데이터 보강 프레임워크

이 그림은 전체 프레임워크를 메타데이터 보강 관점에서 다시 표현한 것입니다.

펌웨어 바이너리가 import되면 디컴파일러와 정적 분석기를 통해 프로그램 표현과 메타데이터가 만들어집니다. 여기서 메타데이터는 함수, 참조, 호출 관계처럼 이후 분석에 사용되는 정보입니다.

본 연구의 큐브샛 에뮬레이터는 같은 펌웨어를 실행하면서 동적 제어 흐름을 기록합니다. 그리고 이 기록을 기존 메타데이터에 반영하면, 이후 보강된 메타데이터를 기반으로 더 넓은 범위의 분석을 수행할 수 있습니다. 따라서 본 연구의 핵심은 에뮬레이션 자체보다, 에뮬레이션에서 얻은 실행 정보를 정적 분석 도구가 재사용할 수 있는 형태로 저장한다는 점입니다.
